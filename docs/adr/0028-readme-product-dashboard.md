# ADR-0028: README を製品入口兼状態ダッシュボードとして扱う

- 状態: 採択
- 日付: 2026-07-09

## 背景

Nape Gesture は CLI 補助コマンドを多数持つが、完成形は通常 GUI アプリである。
README の先頭がコマンド一覧中心になると、初見の読者や後続エージェントが、何を起動すればよいか、何が完成済みか、どこに証跡があるかを追いにくい。

また、GUI、権限導線、完成判定、実機証跡の方針が会話や PR 本文だけに残ると、次回以降の README 更新で同じ品質基準を再現しにくい。

## 決定

- README は製品入口兼状態ダッシュボードとして扱う。
- README の先頭には、プロダクト目的、通常 GUI アプリとしての起動導線、現在の完成状態、未完了または人間作業待ちの項目を置く。
- ユーザーが見る挙動、GUI、権限導線、検証手順、完成状態、配布手順が変わる PR では、README を更新する。更新不要の場合は PR 本文で理由を明記する。
- 長い CLI 例は折りたたみや表で整理し、README の主導線を GUI と完成状態から逸らさない。
- Mermaid は、入力経路、runtime、証跡フローなど、文章だけでは追いにくい構造を示す場合に積極的に使う。
- アプリのスクリーンショットやキャプチャを README に置く場合は、実際に起動したアプリから取得したものだけを使う。未取得の画面をプレースホルダー画像や想像図で代替しない。
- 完成判定、性能基準、PR レビュー方針の正本は docs 側に置き、README は要点とリンクを持つ。
- README で未検証事項を完了済みと表現しない。`need:human` は TCC 操作、物理デバイス操作、証明書操作など、人間が実作業しないと進められない項目に限定する。

## 影響

- PR レビューでは、ユーザー向け変更に対して README 更新または更新不要理由を確認する。
- README は短い紹介文ではなく、最新の利用入口、完成状態、検証入口を示す文書として継続的に更新する。
- GUI キャプチャを追加する場合は、キャプチャ取得手順または取得元 artifact を PR に残し、見た目だけの画像追加にしない。
- Mermaid 図を更新した場合も、実装や docs と矛盾しないかレビュー対象にする。
