# ADR-0004: メインスレッドとサブエージェントの役割分担、PR レビュー、merge 判断

- 状態: 採択
- 日付: 2026-07-08

## 背景

複数のサブエージェントが同じコードベースで作業するため、役割分担と merge 判断が曖昧だと、他者の変更を戻したり、検証不足の PR を進めたりするリスクがある。
メインスレッドは全体ゴールと証跡を維持し、サブエージェントは担当 Issue の範囲で前線を進める必要がある。

## 決定

- メインスレッドは、Issue 整理、PR レビュー、CI とローカル検証の証跡確認、merge 判断、完成判定を担当する。
- サブエージェントは、割り当てられた Issue と明示された所有範囲だけを担当する。
- サブエージェントは、所有範囲外のファイルを編集する必要がある場合、理由を PR 本文または最終報告に明記する。
- サブエージェントは、他者の変更を戻さない。競合がある場合は、既存変更を読んだうえで共存させる。
- コード編集を伴うサブエージェントは、原則として Issue ごとの専用 `git worktree` と `codex/issue-XXX-*` ブランチで作業する。
- メインスレッドの checkout は、PR レビュー、CI 確認、merge 後の `main` 同期に使う。サブエージェントにメイン checkout を直接共有させない。
- 複数サブエージェントが並行する場合は、所有ファイルだけでなく worktree も分ける。同じ checkout を共有すると、branch 切り替えや未コミット差分の取り合いが起きるためである。
- worktree を作った場合、Issue コメントまたは PR 本文に worktree path と branch を残す。
- PR は原則 draft で作成または更新し、メインスレッドがレビュー後に ready 化する。
- PR 本文には、対応 Issue、変更内容、変更ファイル、所有範囲外の変更理由、検証結果、未検証事項を記載する。
- PR レビューでは `docs/pr-review-checklist.md` と関連 ADR を使い、仕様逸脱、入力安全性、テスト不足、証跡不足を優先して確認する。
- GitHub が self-approval を拒否する場合は、approve ではなく review comment として確認内容、ローカル検証、CI 結果、未検証事項を残す。
- merge 判断はメインスレッドが行う。サブエージェントは自分の判断だけで merge しない。
- CI 失敗、必須検証の未実施、未説明の所有範囲外変更がある PR は merge しない。
- docs/config のみの PR では、Swift build を必須にせず、`git diff --check`、YAML parse、リンク整合性など変更対象に合った検証で判断できる。

## 影響

- メインスレッドは実装を抱え込みすぎず、レビューと統合判断に集中できる。
- サブエージェントは担当範囲を明確に保ち、並列作業の衝突を減らせる。
- draft PR のまま更新しても、ready 化と merge はメインスレッドの明示判断に残る。
